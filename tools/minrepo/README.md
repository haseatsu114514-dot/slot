# タイホウ亀島店 設定狙い分析ツール（min-repo データ）

[みんレポ (min-repo.com)](https://min-repo.com/) が公開しているタイホウ亀島店の日次スロットデータを取得・蓄積し、
「設定が入ってそうな場所・法則性」を自動で分析してレポートにするツール。依存パッケージ無し（Node 18+ のみ）。

分析できるもの:

- **イベント日仮説の検証** — 3のつく日 / 4のつく日 / 7のつく日 / 第1土曜日 などの差枚・稼働の通常日比（Δ）、
  月次トレンドと「直近でまだ機能しているか」の判定（→ *4の日を今もやってるか* が分かる）。勝率・p値は各表の「詳細」折りたたみ内
- **日にち別（1〜31日）・曜日別**のヒートマップ
- **末尾分析** — イベント日ごとの末尾成績、日付末尾一致（13日の末尾3 等）、ゾロ目
- **機種シリーズ別** — 北斗・東京喰種・カバネリ・ゴッド・沖ドキ・ジャグラー・ハナハナ × イベント日、年別推移
  （→ *昔は7の日に北斗* の噂を年別で検証できる）
- **全機種の強さランキング** — ウォッチリスト外も含む全機種を平均差枚順に集計（サンプル数・直近60日の再集計つき。
  → *グールとカバネリ以外に強い台はないか* が分かる）
- ウォッチリストのシリーズで**設置が確認できないもの（撤去済み等）はレポートから自動で非表示**（注記だけ出る）
- **稼働（G数）の観察** — 差枚・勝率に加えて平均G数の uplift を日にち別/曜日別/イベント日/末尾/機種別すべてで併記
  （→ *高稼働×高勝率 = 客に信じられていて実際に出る日* / *高稼働×低勝率 = 客が集まるだけの回収日* を見分けられる）
- **凹み台の救済傾向** — 直近7日合計差枚バケツ別の当日勝率（イベント日/通常日で比較）
- **新台・増台・減台・撤去の自動検知** — 設置構成の変化から検出し、導入直後の扱いも集計
- **キャラ誕生日・記念日** — config の仮説リスト + シリーズ×(月,日) の自動スキャン

## 使い方

```sh
# 初回: 過去 12 か月ぶんを取得（config.mjs の backfillMonths で変更可）
# ※分析に使うのは直近 6 か月だけ（config.mjs の analysisMonths）。古い営業方針を引きずらないため。
#   昔の法則（例: 7の日北斗）を検証したいときは analysisMonths: null にして再分析する
node tools/minrepo/run.mjs --backfill

# 量が多いので分割するなら（再実行すると続きから取得する）
node tools/minrepo/run.mjs --backfill --limit 60

# 2回目以降（毎日 or 数日おき）: 新しい日だけ取得して再分析
node tools/minrepo/run.mjs

# 「狙い日」計算の基準日を指定してレポートだけ作り直す（例: 明日向け）
node tools/minrepo/run.mjs report --date 2026-06-13
```

レポート（`data/report.html`）は**単一ファイルのダッシュボード**（スマホ対応・固定ナビ・結論サマリー・
「今後2週間の狙い日」・表のヘッダクリックでソート）。LINE や AirDrop で自分のスマホに送ればそのまま開ける。

結果は `tools/minrepo/data/` に出る:

| ファイル | 内容 |
|---|---|
| `report.html` | ブラウザで開く本体レポート（色付きテーブル） |
| `report.md` | 上位の発見だけのダイジェスト |
| `days.csv` / `models.csv` / `units.csv` / `suffixes.csv` | 自分で Excel 等で分析する用の生データ |
| `analysis.json` | 全分析結果（レポートに出ない詳細含む） |
| `raw/` | 取得した HTML のキャッシュ（パーサ改良後に `run.mjs parse` で再解釈できる） |

仮説の追加・変更は `config.mjs` を編集（イベント日・注目機種・誕生日・凹みの閾値）。
取得し直さなくても `node tools/minrepo/run.mjs parse && node tools/minrepo/run.mjs analyze && node tools/minrepo/run.mjs report` で再計算できる。

### 自動実行（できる限り自動で）

自宅の PC で定期実行する。**クラウド（GitHub Actions 等）からは実行しない**こと（下記の注意参照）。

- macOS / Linux（cron）: `crontab -e` に
  `30 6 * * * cd /path/to/slot && /usr/bin/node tools/minrepo/run.mjs >> tools/minrepo/data/cron.log 2>&1`
- Windows（タスクスケジューラ）: 「基本タスクの作成」→ 毎日 6:30 →
  プログラム `node` / 引数 `tools\minrepo\run.mjs` / 開始場所はリポジトリのフォルダ

min-repo の記事は毎日 1〜4 時頃（前日分）に投稿されるので、朝に回すのがよい。

## 重要な注意

- **自宅回線で実行すること。** min-repo はデータセンター系 IP からのアクセスに *ダミー値*（差枚 0/±1・出率 100% 張り付き）を
  返すことがある。本ツールはこれを検知すると即中断する（ゴミデータ蓄積防止）。中断メッセージが出たら環境を変えること。
- **マナー**: リクエスト間隔は既定 2 秒 + ジッタ。短くしないこと。バックフィルは `--limit` で数日に分けると行儀がよい。
- **私的利用の範囲で。** データは min-repo の独自調査値（推定値）であり、レポートや CSV を再配布・転載しないこと。
  このリポジトリを GitHub Pages 等で公開している場合も `tools/minrepo/data/` は公開に含めない（.gitignore 済み）。
- 機種別詳細（`?kishu=`）はリクエスト数が増える。既定はウォッチリスト該当機種のみ・毎日取得
  （`config.mjs` の `kishuDetail.days: "event"` にするとイベント日+土日だけになり大幅に減る。`--no-kishu` で無効化）。
- 古い記事（おおよそ 2020 年以前）はページ構成が違い、タイトルの「総差枚/平均差枚」だけ取り込まれる。

## 新台・増台の情報源について

- 過去の入替は本ツールが**データから自動検知**する（`report.html` の「新台・増台・撤去の検知」）。
- これからの入替（導入予定）はデータからは分からないので、
  [P-town の店舗ページ](https://p-town.dmm.com/shops/aichi/5713)（新台機種情報）や店の公式 X を見るのが早い。
  旧イベント日・口コミは [みんパチの店舗ページ](https://minpachi.com/%E3%82%BF%E3%82%A4%E3%83%9B%E3%82%A6%E4%BA%80%E5%B3%B6%E5%BA%97/) にまとまっている。

## 仕組み

1. WordPress REST API（`/wp-json/wp/v2/posts?search=タイホウ亀島`）で記事一覧（日付 → 記事URL）を列挙
2. 各日のレポート HTML を取得して `data/raw/YYYY-MM-DD.html` にキャッシュ
3. テーブル（概況 / 機種別 / バラエティ台番別 / 末尾別）をパースして tidy な CSV / JSON に変換
4. 分析（並べ替え検定・符号反転検定で p値を出す）→ HTML / Markdown レポート生成

```
tools/minrepo/
  run.mjs          エントリポイント（fetch / parse / analyze / report）
  config.mjs       店・仮説・マナーの設定
  lib/parse.mjs    HTML パーサ（純粋関数）
  lib/dataset.mjs  tidy データ化・CSV 出力（純粋関数）
  lib/analyze.mjs  分析エンジン（純粋関数）
  lib/report.mjs   レポート生成（純粋関数）
  lib/wp.mjs       記事一覧の列挙
  lib/http.mjs     行儀のよい fetch
  lib/util.mjs     数値・日付・統計
```

テスト: `node tests/minrepo-parse.test.mjs` と `node tests/minrepo-analyze.test.mjs`
（後者は既知の法則を仕込んだ合成データで、分析が法則を検出できるかを確かめる）。

## 読み方のコツ

- 表は**平均差枚と G数（稼働）**が前面。差枚は万枚事故などの外れ値に引っ張られるので、怪しいときは各表の「詳細」内の勝率・p値で裏を取る。
- G数（稼働）は客側の行動。高稼働=高設定の証明ではないが、イベント日が客に信じられているかの傍証になる。
  稼働だけ高くて勝率が伸びない日は「バレてるだけの回収日」の疑い。
- p値が 0.05 を切っていても、仮説をたくさん同時に見ているので 20 回に 1 回は偶然でもそうなる。
  「uplift の大きさ」「月次トレンドの安定感」「n」をセットで判断すること。
  反復数（サンプル数）は `config.mjs` の `analysis.iterations`（既定 4000）で増やせる。
- 店が「データサイトに載る数字」を意識して営業を変える可能性もある。過去の法則は未来を保証しない。
